title: Sessions

A session is **one conversation** with the agent. Each session gets a
**fresh sandbox VM** with your project repo cloned and checked out on a
branch named after the session ID. The sandbox is ephemeral; the branch
persists. Anything the agent commits and pushes lands as a real branch
on your project repo — review it, open a PR, merge into main when it's
ready.

## Lifecycle

```
queued → branching → provisioning → running → ended
```

| Stage          | What's happening                                          |
| -------------- | --------------------------------------------------------- |
| `queued`       | Accepted, waiting for a slot.                             |
| `branching`    | Platform is creating the session branch.                  |
| `provisioning` | The sandbox provider is spinning up the VM.               |
| `running`      | Live. The dashboard tunnels into the daemon over HTTPS.   |
| `ended`        | Sandbox stopped. Branch and history stick around.         |

Concurrent-session caps are enforced per account. If you hit your
tier's cap, new-session requests return `429`.

## Branch model

- The session branch is **named after the session UUID** by construction.
  `KORTIX_SESSION_ID` and `KORTIX_BRANCH_NAME` are interchangeable.
- The session branches from the project's default branch (usually `main`)
  at session-create time.
- Triggers create their own session branches the same way an interactive
  session would — there's no "background" mode that mutates `main`
  directly.

## What survives

**Only what you commit and push.** The sandbox filesystem disappears
when the session ends. That's the whole point — you can `rm -rf`,
install whatever, run unsafe scripts; it's disposable.

In practice: have the agent commit small, meaningful chunks throughout
the session, not one giant commit at the end. If the sandbox dies
mid-run, you keep everything that was already pushed.

## Layout inside the sandbox

```
/workspace                  ← WORKDIR + HOME. The project repo lives here directly.
/workspace/.kortix/         ← Repo-internal Kortix folder (Dockerfile + opencode dir).
/usr/local/bin/kortix-agent ← The sandbox daemon (supervisor + reverse proxy).
/usr/local/bin/kortix-entrypoint ← The container's ENTRYPOINT.
```

Since `HOME=/workspace`, tool caches (`~/.npm`, `~/.cache`, `~/.bun`)
land alongside your repo files. That's expected. They're in the
session's `.gitignore` and never get committed.

## Environment variables

The platform injects these at session boot, in addition to whatever
`[env]` secrets you declared:

| Variable                          | What                                                          |
| --------------------------------- | ------------------------------------------------------------- |
| `KORTIX_PROJECT_ID`               | UUID of this project.                                         |
| `KORTIX_SESSION_ID`               | UUID of this session. Also the branch name.                   |
| `KORTIX_BRANCH_NAME`              | Same as `KORTIX_SESSION_ID` — what your work pushes to.       |
| `KORTIX_REPO_URL`                 | HTTPS clone URL for the project repo.                         |
| `KORTIX_DEFAULT_BRANCH`           | The repo's default branch (usually `main`).                   |
| `KORTIX_BASE_REF`                 | The ref the session branched off of.                          |
| `KORTIX_WORKSPACE`                | `/workspace` by default.                                      |
| `KORTIX_PROJECT_TARGET`           | Path the repo is cloned to (equal to `KORTIX_WORKSPACE`).     |
| `KORTIX_SERVICE_PORT`             | `8000` — the daemon's external port (proxies opencode).       |
| `KORTIX_AGENT_NAME`               | The opencode agent the session was created with.              |
| `KORTIX_INITIAL_PROMPT`           | Set when the session was spawned by a trigger.                |
| `KORTIX_GITHUB_TOKEN`             | Token git uses to push your commits back to origin.           |
| `KORTIX_LLM_BASE_URL` / `_TOKEN`  | Where to route LLM calls (for routed billing).                |

The `KORTIX_*` prefix is reserved for platform variables.
[Declaring a user secret with that prefix is rejected.](/docs/secrets)

## Pushing from a session

Git is pre-configured with `KORTIX_GITHUB_TOKEN`. The agent can
`git push origin HEAD` without any additional setup — the push goes to
the session branch on the project repo. From there, review and merge
happens via the dashboard or GitHub directly.

## The daemon's control surface

A small binary (`kortix-agent`) runs as the container entrypoint and
fronts opencode on `KORTIX_SERVICE_PORT`. It exposes:

| Path                   | Purpose                                                          |
| ---------------------- | ---------------------------------------------------------------- |
| `GET /kortix/health`   | Always returns 200. Reports daemon + opencode state.             |
| `POST /kortix/refresh` | Signed fast-forward of the session branch + opencode restart.    |
| `/*`                   | Reverse-proxied to opencode. `503` while opencode boots.         |

`/kortix/refresh` is how the dashboard applies an out-of-band change
(e.g., a manifest edit committed from a parallel session) without
re-provisioning the sandbox.
