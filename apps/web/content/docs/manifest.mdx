`kortix.toml` at the root of your repo is the only file the platform
treats as authoritative. Unknown top-level tables are ignored, so you can
park your own metadata in it without confusing anything.

## Top-level

```toml
kortix_version = 1
```

Schema version. Manifests without it are read as v1. A manifest declaring
a version higher than the platform knows about is rejected outright —
better than silently misreading future fields.

## `[project]`

```toml
[project]
name        = "Company OS"
description = "What this project is."
```

| Field         | Type   | Required | Notes                          |
| ------------- | ------ | -------- | ------------------------------ |
| `name`        | string | yes      | Display name in the dashboard. |
| `description` | string | no       | Short summary.                 |

## `[env]`

Declares the secrets the runtime needs. Values live in the Kortix Secrets
Manager — never inline in the file. The platform decrypts and injects
them as plain environment variables when a session starts.

```toml
[env]
required = ["DATABASE_URL"]
optional = ["ANTHROPIC_API_KEY", "OPENAI_API_KEY"]
```

| Field      | Type     | Notes                                                                    |
| ---------- | -------- | ------------------------------------------------------------------------ |
| `required` | string[] | Session refuses to start if any of these are unset in Secrets Manager.   |
| `optional` | string[] | Surfaced in the UI so users can set them; absence does not block boot.   |

Names must match `[A-Z_][A-Z0-9_]{0,63}`. The `KORTIX_*` prefix is
reserved for platform variables — declaring one is rejected by the
validator.

## `[sandbox]`

Points at the Dockerfile that becomes the base for session sandboxes.
The Kortix runtime layer (opencode CLI, the agent daemon, the entrypoint)
is layered on top automatically.

```toml
[sandbox]
dockerfile = ".kortix/Dockerfile"
context    = "."
```

| Field        | Type   | Default              | Notes                                  |
| ------------ | ------ | -------------------- | -------------------------------------- |
| `dockerfile` | string | `.kortix/Dockerfile` | Repo-relative. Absolute / `..` rejected. |
| `context`    | string | `.`                  | Build context root, repo-relative.     |

See **[Sandbox image](/docs/dockerfile)** for what the Kortix layer
injects and what your Dockerfile can / cannot do.

## `[opencode]`

Where opencode's config dir lives. The agent daemon launches opencode
with `OPENCODE_CONFIG_DIR` pointed at this folder.

```toml
[opencode]
config_dir = ".kortix/opencode"
```

| Field        | Type   | Default            | Notes                                            |
| ------------ | ------ | ------------------ | ------------------------------------------------ |
| `config_dir` | string | `.kortix/opencode` | Repo-relative. Absolute / `..` rejected.         |

Inside that directory you put agents, skills, slash commands, tools,
and `opencode.jsonc` — the same layout opencode reads by convention.
See [opencode.ai/docs](https://opencode.ai/docs/) for what each file
does; Kortix doesn't read them, opencode does.

## `[[triggers]]`

Cron + signed-webhook entries that spawn fresh sessions. Full reference
in **[Triggers](/docs/triggers)**.

```toml
[[triggers]]
slug    = "daily-digest"
type    = "cron"
cron    = "0 0 9 * * 1-5"
prompt  = "Summarize yesterday's commits."
```

| Field        | Required          | Notes                                            |
| ------------ | ----------------- | ------------------------------------------------ |
| `slug`       | yes               | URL-safe, unique per project.                    |
| `type`       | yes               | `"cron"` or `"webhook"`.                         |
| `prompt`     | yes               | Mustache-templated body sent to the agent.       |
| `agent`      | no (default)      | Agent name from your opencode config dir.        |
| `enabled`    | no (`true`)       | Disable without deleting.                        |
| `cron`       | cron only         | 6-field croner: `s m h day month weekday`.       |
| `timezone`   | cron only (`UTC`) | IANA tz, e.g. `"America/Los_Angeles"`.           |
| `secret_env` | webhook only      | Name of a `project_secrets` entry (HMAC secret). |

## `[[apps]]` (experimental)

Gated by the platform flag `KORTIX_APPS_EXPERIMENTAL`. When off, entries
are parsed but never deployed. One `[[apps]]` entry per deployable
surface (think `fly.toml`-style):

```toml
[[apps]]
slug      = "marketing-site"
enabled   = true
domains   = ["marketing.example.com"]
framework = "next"

  [apps.source]
  type      = "git"            # or "tar"
  repo      = "https://github.com/me/site"
  branch    = "main"
  root_path = "apps/site"

  [apps.build]
  command = "pnpm build"
  out_dir = "dist"

  [apps.env]
  NEXT_PUBLIC_API_URL = "https://api.example.com"
```

If apps are disabled on the platform, the dashboard hides the section
and the manifest entries are inert.

## Round-trip rules

When the dashboard edits the manifest (adding a trigger, flipping
`enabled`), it's a read-modify-write on the same file. To keep the diff
clean across UI + in-session edits:

- Keep `kortix_version` as the first key.
- Inside a `[[triggers]]` entry, write fields in this order: `slug`,
  `name`, `type`, `agent`, `enabled`, then type-specific fields, then
  `prompt` last.
- If you add a webhook trigger before its secret is set, declare the
  secret name in `[env].optional` so it shows up in the Secrets Manager,
  and leave the trigger `enabled = false` until the value is in.
