---
title: Kortix vs runtime config
description: The strict ownership boundary between kortix.yaml and harness-native runtime config.
---

A Kortix project's config has two owners, and the boundary is strict.

- **Kortix owns** the manifest (`kortix.yaml`) and `.kortix/Dockerfile` — what a project is, its sandbox, secrets, triggers, runtime routing, and which agents the platform may launch/authorize.
- **The selected harness owns** its native config directory — how that runtime behaves. Claude Code, Codex, OpenCode, and Pi keep their own prompts, tools, model/provider settings, permissions, and local conventions in the format they already understand.

In v3, `kortix.yaml` declares named `runtimes:`. Each runtime selects a `harness` and optional `config_dir`; the sandbox daemon maps that directory to the harness-native env var (`CLAUDE_CONFIG_DIR`, `CODEX_HOME`, `OPENCODE_CONFIG_DIR`, or `PI_CODING_AGENT_DIR`) and speaks ACP to the harness process.

In `kortix.yaml`, `agents:` is a map keyed by logical Kortix agent name. Each agent chooses a `runtime`, optional harness-native `agent` name, and platform governance: `connectors`, `secrets`, `kortix_cli`, `skills`, `workspace`, `enabled`. Behavioral concerns stay in the selected harness's native files.

`agents:` is deny-by-default in `kortix.yaml`: an agent with no `connectors`/`secrets`/`kortix_cli`/`skills` key gets none of them. Any logical agent not declared in the map is fully denied and cannot be launched through the platform, even if a native harness file exists in the repo.

<Callout type="info" title="Legacy: kortix.toml">
v1 projects (`kortix.toml`) express the same governance idea with `[[agents]]` — an array of tables instead of a name→block map, secrets are called `env` instead of `secrets`, and defaults run the other way: an omitted field on a declared agent falls back to permissive per-field defaults rather than "none," and a project with **no** `[[agents]]` section at all skips governance entirely (every agent inherits the launching user's full role). See [kortix.toml](/docs/reference/manifest) for the v1 field table.
</Callout>

## Ownership table

| Concern | Lives in | Read by |
| --- | --- | --- |
| Project metadata (`name`, `description`) | `kortix.yaml` `project:` | Kortix |
| Env / secrets spec | `kortix.yaml` `env:` | Kortix |
| Sandbox base image | `kortix.yaml` `sandbox:` + `.kortix/Dockerfile` | Kortix |
| Runtime profiles | `kortix.yaml` `runtimes:` | Kortix |
| Runtime config directories | `kortix.yaml` `runtimes.<name>.config_dir` | Kortix → selected harness |
| Triggers (cron + webhook) | `kortix.yaml` `triggers:` | Kortix |
| Platform-launchable agents and grants | `kortix.yaml` `agents:` | Kortix |
| Channels | Not in the manifest — dashboard/CLI-managed (`kortix channels`). The connector itself (`provider: channel`) is still declared under `connectors:` once you connect one. | Kortix (live routing is the `chat_channel_bindings` table) |
| Agent personas | Harness-native config dir | Selected harness |
| Skills / commands / tools / plugins | Harness-native config dir | Selected harness |
| MCP servers | Harness-native config dir | Selected harness |
| Model / provider config | Harness-native config dir or Kortix LLM gateway policy | Selected harness; product model pickers should come from the server / LLM gateway catalog |

<Callout type="info" title="Rule of thumb">
If it describes the project, the sandbox, when work runs, what gets deployed, or what the platform is allowed to launch/authorize, it belongs in `kortix.yaml`. If it describes how the selected harness executes, it belongs in that harness's native config directory.
</Callout>

Full manifest field reference: [the manifest](/docs/reference/manifest). For behavior authoring, use the selected harness's own documentation.

## Where the agent runs

The agent runtime runs in the Kortix cloud sandbox. The sandbox agent server launches the selected ACP harness with the compiled runtime profile and config dir. There is no Kortix command that runs the cloud session agent on your machine; the [CLI](/docs/reference/cli) is a cloud control plane, not a local runner.

Because behavior stays harness-native, files remain portable to the harness that owns them. For example, an OpenCode runtime profile can still use `.kortix/opencode/`, while a Codex profile can use `.codex/`.

New runtime config files reach future sandbox sessions only after a [change request](/docs/reference/change-requests) merges them to `main`.
