This page lists the REST routes, scopes, and rotation behavior for project secrets. For the secret model and manifest grants, see [Secrets](/docs/concepts/secrets).

## Scopes

Every secret has a `scope`. Default: `runtime`.

- **Runtime scope** — the secret injects into the sandbox as an environment variable. Kortix uses this scope for most secrets.
  - **Shared** — the default. The value applies to every session in the project. Every project member with read access sees the shared value.
  - **Personal override** — a member's own value for one secret. It replaces the shared value only in that member's sessions.
- **Connector scope** — the secret holds a credential for a connector. See [Connector scope](#connector-scope) below.

Kortix never lets you set a personal override on an LLM provider credential, for example `ANTHROPIC_API_KEY`. The gateway always reads the shared value for these keys.

Each secret also has two names:

- `identifier` — the name you list in an agent's `secrets` grant. Format: `^[A-Za-z0-9][A-Za-z0-9_.-]{0,127}$`.
- `name` — the environment variable name injected into the sandbox. Format: `^[A-Z_][A-Z0-9_]{0,63}$`.

In most projects `identifier` and `name` are the same string.

Reading or writing a secret needs the `project.secret.read` or `project.secret.write` permission. A custom project role can withhold either permission, even from a manager.

## REST routes

All routes sit under `/v1/projects/{projectId}`.

| Method | Path | Description |
|---|---|---|
| GET | `/secrets` | List secrets. Scoped to the caller's grant if the caller is a scoped agent token. |
| POST | `/secrets` | Create or update the shared value. Body: `{name, identifier?, value}`. |
| DELETE | `/secrets/{name}` | Delete the shared value. Personal overrides stay in place. |
| PUT | `/secrets/{name}/personal` | Set or turn on the caller's personal override. |
| DELETE | `/secrets/{name}/personal` | Remove the caller's personal override. |

`POST /secrets` rejects names that start with `KORTIX_` — that prefix is reserved for the platform. It returns `409` if the `identifier` already exists with a different `name`.

## Rotation and propagation

A secret write does not wait for a session restart. Kortix pushes the change to every active sandbox in the project:

1. Kortix builds a new environment snapshot for the sandbox, using the running agent's `secrets` grant.
2. The sandbox writes the snapshot to the live agent environment. New tool calls pick up the change right away.
3. If the changed secret is an LLM provider credential, Kortix also restarts the sandbox's `opencode` process. The next prompt in the session uses the new credential.

This push is best-effort. The API call that changes the secret returns before the push finishes. A failed push is only logged, not retried. A sandbox with a failed push keeps the old value until the next successful push, or until the session restarts.

## Connector scope

A secret with `scope: connector` holds a credential for a connector, for example an OAuth token. Kortix resolves connector-scope secrets on the server, inside the connector executor. It never injects a connector-scope secret into a sandbox environment, and `GET /secrets` does not list them.
