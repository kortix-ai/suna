title: ACP bridge
description: The in-sandbox daemon's raw ACP endpoint — routes, JSON-RPC semantics, SSE replay, status codes, and process lifecycle.

Every session sandbox runs the Kortix daemon, and the daemon exposes a raw
[Agent Client Protocol](https://agentclientprotocol.com) bridge under `/acp`.
The bridge relays unmodified JSON-RPC 2.0 envelopes between clients and one
harness process per server id — it does not translate, filter, or re-shape the
protocol. All `/acp` routes require the signed `X-Kortix-User-Context` header;
most integrations should use `@kortix/sdk`'s
[ACP surface](/docs/sdk/acp) instead of calling the bridge directly.

## Routes

| Method | Path | Behavior |
| --- | --- | --- |
| `GET` | `/acp/` | List live ACP servers. |
| `POST` | `/acp/:serverId` | One unmodified JSON-RPC 2.0 envelope per request. The first `POST` for a new `serverId` must carry `?agent=<harnessId>` and lazily spawns exactly one harness process. Responses: `200` JSON for a request/response, `202` empty for notifications and client-to-agent responses, `409` if the `serverId` already runs a different harness, `415` for the wrong content type, `400` for an unknown agent or a malformed envelope, `502` on an upstream process error. |
| `GET` | `/acp/:serverId` | SSE stream of agent-originated envelopes (`id: <n>` + `data: <json>` frames). A `Last-Event-ID` header replays buffered events after that sequence. `404` for an unknown server, `406` if `Accept` excludes `text/event-stream`, `400` for a bad `Last-Event-ID`. Keepalive comment every 15s. |
| `DELETE` | `/acp/:serverId` | Terminates the harness process. Idempotent — returns `204`. |

## Process lifecycle

The first `POST /acp/:serverId?agent=<harness>` lazily starts exactly one
pinned ACP process for that server id; every later request against the same
`serverId` reuses the already-running process instead of spawning another
one. Sending a different `agent` for a `serverId` that's already running a
harness does not restart or switch it — the bridge rejects the request with
`409`. `DELETE` terminates the harness process and is idempotent: calling it
again after the process is gone still returns `204`.

A write queue serializes stdin writes to the harness process, but it does not
serialize request lifetimes. A long-running `session/prompt` call that's
still streaming its response never blocks the bridge from writing and
answering an agent-originated permission request that arrives in the
meantime — the queue orders the bytes going in, not how long any one
in-flight call takes to resolve.

## Streaming and replay

`GET /acp/:serverId` opens an SSE stream of agent-originated envelopes. Each
frame carries an `id: <n>` line with an increasing sequence number followed
by a `data: <json>` line with the envelope itself. A client that reconnects
after a drop can send a `Last-Event-ID` header with the last sequence number
it saw, and the bridge replays every buffered event after that sequence
before resuming the live stream. The connection sends a keepalive comment
every 15 seconds to hold the stream open through idle periods and
intermediary timeouts.

The SSE buffer that makes replay possible lives in the sandbox and is
bounded — it does not grow without limit, and it does not survive the
harness process or sandbox going away. Durable history of the same envelopes
lives in the platform's [envelope log](/docs/reference/session-runtime),
which is the source of truth for anything beyond what the in-sandbox buffer
can still replay.

## Environment

Harness processes spawned by the bridge inherit only the sandbox environment
plus the filtered project-secret snapshot — nothing else is injected into
the process at spawn time.

<Callout type="info" title="Overriding the ACP process">
For controlled rollouts and testing, the sandbox daemon honors
`KORTIX_ACP_<HARNESS>_PATH` (replacement launch command) and
`KORTIX_ACP_<HARNESS>_ARGS` (a JSON string array, never shell-parsed) for
`CLAUDE`, `CODEX`, `OPENCODE`, and `PI`. See
[Harnesses](/docs/reference/harnesses) for the harnesses these apply to.
</Callout>
