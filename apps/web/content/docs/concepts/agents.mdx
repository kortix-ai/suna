An agent in Kortix is a logical project entry. It chooses a runtime profile,
declares governance grants, and optionally points at a harness-native agent name.
The runtime profile speaks ACP, so the same session UI can drive Claude Code,
Codex, OpenCode, or another supported harness.

Harness-native behavior stays in that harness's normal config directory:

- Claude Code uses `CLAUDE_CONFIG_DIR`.
- Codex uses `CODEX_HOME`.
- OpenCode uses `OPENCODE_CONFIG_DIR` (default `.kortix/opencode`).
- Pi uses `PI_CODING_AGENT_DIR`.

Kortix does not translate those native behavior files. It compiles `kortix.yaml`
to a launch plan and passes the selected config dir to the harness.

## The manifest is routing and governance

In v3, `kortix.yaml` has two relevant maps:

- `runtimes:` defines named ACP runtime profiles (`harness` + optional
  `config_dir`).
- `agents:` defines logical Kortix agents. Each agent chooses a `runtime`, can
  set a harness-native `agent` name, and declares platform governance:
  `connectors`, `secrets`, `kortix_cli`, `skills`, `workspace`, `enabled`.

`default_agent` is required at the top level and must name a declared, enabled
agent.

v3 is **deny-by-default**: an agent with no `connectors`/`secrets`/`kortix_cli`/
`skills` key gets none of that access. Give it `all` explicitly to grant full
access. Chat inputs, trigger pickers, and channel pickers list registered agents
from the API, not from a sandbox-local harness query — this doesn't mean every
native harness agent file must be registered; unregistered files can still exist
for local experiments or runtime internals.

<Callout type="info" title="Legacy kortix.toml">
v1 projects declare the same grants as an array, `[[agents]]`, with the secrets
field named `env` instead of `secrets`, and per-field defaults instead of
deny-all. A v1 project with no `[[agents]]` at all runs in legacy unrestricted
mode — an agent inherits the launching user's full role. Once `[[agents]]`
exists, an unlisted agent is default-denied, same as v2.
</Callout>

Every project ships a default agent + the kortix-system skill, so it runs with no
setup. Beyond reading/writing repo files, an agent can use external tools through
connected integrations (per project — it only uses what's connected; see
[Connecting your tools](/docs/guides/connecting-tools)).

New agents, skills, or tools reach future sessions only after a
[change request](/docs/concepts/change-requests) merges them. For native
behavior authoring, use the selected harness's own documentation.
