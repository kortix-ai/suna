---
title: Secrets
description: How Kortix stores and injects encrypted credentials into each session.
---

A secret is a per-project credential — an API key, token, or connection
string — that a session needs but that must not live in the repository.
Kortix stores secrets on the project, never on the account.

## Two names per secret

Each secret has an identifier and a name. The identifier is the handle you
and an agent's `secrets` grant use. The name is the uppercase
environment-variable key injected into the sandbox, for example
`STRIPE_API_KEY`. In most cases the identifier and the name match. They
differ only when a project holds several candidate values for one key. For
example, a primary and a backup Google Maps key can both resolve to
`GOOGLE_MAPS_API_KEY`.

## Shared and personal scope

A secret is either shared or a personal override:

- **Shared** — the project-wide value. Every project member with read
  access sees every shared secret. Kortix has no per-member sharing control
  for a single secret.
- **Personal override** — your own value for one name, used instead of the
  shared row for sessions you start. Today Kortix uses this only for one
  OAuth login credential.

Kortix never lets an LLM provider key (for example `ANTHROPIC_API_KEY`)
become a personal override. The model gateway only reads the shared row.

## Runtime injection

At session start, Kortix resolves each granted secret name for the
launching user. It uses your active personal override if you have one,
otherwise the shared value, and injects the result as a plain environment
variable. Names starting with `KORTIX_` are reserved for the platform and
never accepted as secret names.

Setting or removing a secret propagates to every running session for the
project on a best-effort basis. If the changed secret affects the model
provider, Kortix also restarts the agent process in the sandbox.

## Agent access

An agent only receives the secrets its manifest grants. In `kortix.yaml`,
the agent's `secrets` field lists identifiers, or is empty to deny all —
the default when you omit it. See [Agents](/docs/concepts/agents).

## Connector secrets

A secret with scope `connector` is different: it authenticates a
[connector](/docs/concepts/connectors), not a session. Kortix resolves it
server-side and never injects it into a sandbox.

<Callout type="info" title="Encryption">
Kortix encrypts every secret value with AES-256-GCM, using a key derived
per project from the platform master key.
</Callout>

To add or rotate secrets, see
[Managing secrets](/docs/guides/managing-secrets). For the full field and
route reference, see [Secrets reference](/docs/reference/secrets).
