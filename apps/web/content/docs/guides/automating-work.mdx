A [trigger](/docs/concepts/triggers) starts a [session](/docs/concepts/sessions) with no person present, on a schedule or from an inbound webhook. This guide sets up one of each with the CLI.

## Set up a cron trigger

<Steps>

<Step>
### Add the trigger

```sh
kortix triggers add daily-digest --type cron \
  --cron "0 0 9 * * 1-5" --timezone America/Los_Angeles \
  --prompt "Summarize yesterday's activity and save it as a daily note."
```

`cron` is a 6-field expression: second, minute, hour, day, month, weekday. This command edits your local `kortix.yaml` only.
</Step>

<Step>
### Ship it

```sh
kortix ship
```

`kortix ship` commits `kortix.yaml` and pushes it. The schedule goes live once this lands on your project's default branch.
</Step>

<Step>
### Confirm it runs

```sh
kortix triggers ls
```

The list shows each trigger's slug, state, and when it last fired. To fire it now instead of waiting for the schedule, run `kortix triggers fire daily-digest`.
</Step>

</Steps>

## Set up a webhook trigger

A webhook trigger needs a secret. Kortix uses it to check the signature on every incoming request.

<Steps>

<Step>
### Add the secret

```sh
kortix secrets set WEBHOOK_SECRET=<a-random-value>
```

See [Managing secrets](/docs/guides/managing-secrets) for more on secrets.
</Step>

<Step>
### Add the trigger

```sh
kortix triggers add new-lead --type webhook \
  --secret-env WEBHOOK_SECRET \
  --prompt "A new lead arrived: {{ body.name }} ({{ body.email }}). Add it to the CRM."
```

`--secret-env` names the secret that signs requests to this trigger.
</Step>

<Step>
### Ship it

```sh
kortix ship
```
</Step>

<Step>
### Send it a request

Kortix builds the webhook URL from your project id and the trigger's slug:

```
POST /v1/webhooks/projects/<project-id>/<slug>
```

Send a signed `POST` request to this URL from the external service. Kortix checks the signature against `WEBHOOK_SECRET`. It then starts a session with the request body available in the prompt. See [Triggers reference](/docs/reference/triggers) for the exact signature header and format.
</Step>

</Steps>

## Next steps

By default, each fire starts a fresh session on a new branch. A trigger can instead reuse or pin a session. A webhook trigger can also filter which payloads start one. See the [triggers reference](/docs/reference/triggers) for session strategy, template variables, and every manifest field.
