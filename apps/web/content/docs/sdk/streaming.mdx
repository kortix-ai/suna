Live updates are the SDK's most powerful surface and its most
environment-sensitive one. This page covers how the stream works, the pattern
for consuming it correctly, and precisely where it runs.

## The canonical stream is ACP envelopes

A session's live stream is **agent-originated ACP envelopes** — the same
JSON-RPC requests/notifications/responses documented on the
[ACP](/docs/sdk/acp) page. `session.stream({ onEvent })` delivers each one as
`{ id, envelope }`; there is no separate "chat event" shape underneath it
anymore. Most hosts don't parse envelopes by hand: the
[`AcpSession`](/docs/sdk/acp#acpsession) store (and `useSession` /
`useAcpSession` in React) reduces the envelope log into `chatItems`,
`pendingPrompts`, `turnState`, and the rest of the snapshot for you. Reach for
the raw envelope stream below only when you need something the reducer
doesn't already expose.

## The mental model

A session's events come from the **ACP runtime inside its sandbox**, reached
through the Kortix API as a proxy. There is no separate WebSocket endpoint or
`EventSource` — the transport is **`fetch` with a streaming response body**
(server-sent events read via `ReadableStream` + `TextDecoderStream`).

The SDK owns everything above the wire: **reconnection, backoff, heartbeat
detection, and event coalescing** live in the SDK's event-stream layer, so a
dropped connection resumes without your code noticing.

## The pattern

Three rules make streaming correct:

1. **Ready the session first.** The runtime doesn't exist until the sandbox is
   provisioned or resumed — that's `ensureReady()`.
2. **Connect the stream before sending**, so no early events are missed.
3. **`send()` already tells you when the turn is done** — no separate "idle"
   signal to wait for.

```ts
const session = kortix.session(projectId, sessionId);
await session.ensureReady();

const stream = await session.stream({
  onEvent: ({ envelope }) => {
    // Every event is an ACP envelope. `session/update` notifications carry
    // incremental agent output — text deltas, tool-call updates, plans, …
    if (!('method' in envelope) || envelope.method !== 'session/update') return;
    const update = (envelope.params as any)?.update;
    if (update?.content?.type === 'text') process.stdout.write(update.content.text);
  },
  onError: (error) => console.error('[ACP stream error]', error),
});

// session.send() drives an ACP prompt turn and resolves with { stopReason }
// once the turn ends — the stream is for incremental rendering, not for
// detecting completion.
await session.send('…');
stream.close();
```

In React, skip all of this: [`useSession`](/docs/sdk/react) owns the stream,
readiness, and reconnect for you, and exposes the reduced `acp.envelopes` /
`acp.chatItems`-shaped state directly. Framework-free hosts that want the same
reduction instead of parsing `session/update` payloads by hand should build on
[`createAcpSession`](/docs/sdk/acp#acpsession) (`subscribe` + `getSnapshot`) or
project a stored envelope log with
[`projectAcpChatItems`](/docs/sdk/acp#projections) rather than switching on
raw envelope methods.

## Readiness is a handshake, not a one-liner

`ensureReady()` issues **one bounded long-poll** against the session's
`/start` endpoint. On a warm session it resolves immediately; on a **cold
boot** — a sandbox being provisioned for the first time — the long-poll can
return before the sandbox is up, and `ensureReady()` throws a typed
`ApiError` with `code: 'RUNTIME_UNAVAILABLE'`.

That error means _not ready yet_, not _failed_. Retry it — each retry
re-attaches to the same server-side provision:

```ts
async function retryUntilReady<T>(ensure: () => Promise<T>): Promise<T> {
  const deadline = Date.now() + 300_000;
  for (;;) {
    try {
      return await ensure();
    } catch (error) {
      const provisioning = error instanceof ApiError && error.code === 'RUNTIME_UNAVAILABLE';
      if (!provisioning || Date.now() > deadline) throw error;
      await new Promise((r) => setTimeout(r, 3_000));
    }
  }
}
```

`ensureReady()` is idempotent and deduplicates concurrent starts for the same
session, so racing calls ride one `/start` instead of issuing several.

## Runtime support matrix

The SSE transport needs `fetch` with a real `ReadableStream` response body
**and** `TextDecoderStream`. On React Native, `streamTransport: 'auto'`
(the default) falls back to durable-transcript polling instead of failing —
`session.stream()` still works, just at poll cadence rather than
token-by-token. In any other environment without SSE support, force
`streamTransport: 'poll'`:

| Runtime                   | SSE      | Notes                                                                                                                                                                                                          |
| ------------------------- | -------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Modern browsers           | ✅       | `TextDecoderStream` needs Safari 16.4+                                                                                                                                                                        |
| Node ≥ 18                 | ✅       | `fetch` and `TextDecoderStream` are global                                                                                                                                                                    |
| Bun                       | ✅       |                                                                                                                                                                                                                 |
| Cloudflare Workers / edge | ✅       |                                                                                                                                                                                                                 |
| **React Native / Expo**   | ❌       | RN's `fetch` has no `response.body`, and Hermes has no `TextDecoderStream`, so the SSE transport can't run there. `session.stream()` still works: `AcpClient` automatically falls back to durable-transcript polling, so live updates keep arriving — at poll cadence, not token-by-token. |

<Callout type="warn">
  Everything *except the SSE transport* — projects, sessions, files, secrets, the whole REST surface,
  and `session.stream()` itself via its polling fallback — works everywhere `fetch` does, including
  React Native. The matrix above is only about which transport carries the live event stream.
</Callout>

## Deprecated: the OpenCode-wire event union

<Callout type="warn" title="narrowChatEvent has been removed">
  Earlier SDK versions exposed `narrowChatEvent(event)` to narrow `session.stream()`'s output into
  the OpenCode-wire union below. That helper is **gone from the SDK's export surface** — it exists
  only in an internal dev playground script, not in `packages/sdk/src`. `session.stream()` delivers
  ACP envelopes exclusively today (see [above](#the-canonical-stream-is-acp-envelopes)); there is no
  supported way to get this union back out of it.
</Callout>

The union itself documented the full chat lifecycle a legacy OpenCode-wire
consumer would see:

| `type`                                          | When                                                                                              |
| ----------------------------------------------- | ------------------------------------------------------------------------------------------------- |
| `message.updated` / `message.removed`           | a message's metadata changed / it was deleted                                                     |
| `message.part.updated` / `message.part.removed` | a part (text, tool call, file, reasoning) grew or was removed — this is the token-by-token signal |
| `session.status`                                | the session's busy/working state changed                                                          |
| `session.idle`                                  | the turn finished                                                                                 |
| `session.error`                                 | the turn failed (LLM error, tool crash) — carries the error payload                               |
| `question.asked` / `question.answered`          | the agent asked for permission/input, and the resolution                                          |

These wire shapes still back a handful of deprecated projections that are kept
working for existing consumers — [`classifyTurn`](/docs/sdk/turns) still turns
raw `MessageWithParts` into an exhaustively-typed union of renderable parts,
and `formatTranscript` still formats them to Markdown — but neither is fed by
`session.stream()` anymore. New code renders the live transcript from ACP
envelopes via [`projectAcpChatItems`](/docs/sdk/acp#projections) instead.
